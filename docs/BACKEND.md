# ezctx — бекенд на устройстве

## Назначение

Извлечение текстового контекста из аудио/видео на самом устройстве (без серверной части) и подготовка его для подачи в LLM. Основная задача: получить полную расшифровку речи, максимально эффективно используя бесплатный тир Groq Whisper.

## Источник идей

Бекенд переносится из существующего проекта `~/projects/LectureLog/` (Python/FastAPI). Конкретно:

- `lecturelog/pipeline/transcribe.py` — пул Groq-ключей + чанкование + параллельная отправка.
- `lecturelog/llm/key_pool.py` — обобщённый round-robin пул с rate-limit защитой.

В ezctx логика переписывается на **Dart** и работает локально внутри Flutter-приложения (без FastAPI-обёртки).

## Транскрибация через Groq

### Используемые модели

- `whisper-large-v3` — качественнее.
- `whisper-large-v3-turbo` — быстрее.
- В UI — переключатель «качество/скорость». Дефолт: `large-v3`.

### Жёсткие лимиты Groq Free Tier (фиксируем как константы)

- **Максимум 19.5 MB на один файл** (берём 19 MB с запасом).
- **Поддерживаемые форматы:** `flac`, `mp3`, `mp4`, `mpeg`, `mpga`, `m4a`, `ogg`, `wav`, `webm`.
- За пределами этого списка — отказ на этапе валидации, без попыток отправки.

### Пул ключей (round-robin + rate-limit)

Порт `GroqKeyPool` из `transcribe.py`:

- Ключи `GROQ_API_KEY` вводит сам пользователь в настройках приложения (несколько штук, по одному в строку или через UI «добавить ключ»). В настройках — кликабельная ссылка на консоль Groq (`https://console.groq.com/keys`), чтобы пользователь мог быстро создать ключи.
- Хранятся в защищённом хранилище устройства (`flutter_secure_storage`). В сборку ключи не зашиваются.
- При 429/503 ключ блокируется. Длительность блокировки берём из заголовков ответа Groq:
  - `retry-after` (секунды) — приоритетно.
  - `x-ratelimit-reset-*` — если есть, парсим время до сброса лимита.
  - Если заголовков нет — фоллбэк 60 сек.
- Пул выдаёт следующий доступный ключ. Если все заблокированы — ожидание до ближайшей разблокировки.
- Учёт RPM на ключ через `last_request + interval` (как в `key_pool.py`).
- **UI:** для каждого ключа показываем статус (активен / заблокирован до HH:MM:SS) и, по возможности, остаток бесплатной квоты на сегодня (`x-ratelimit-remaining-*` из заголовков Groq). Пользователь должен видеть, сколько ещё минут расшифровки доступно прямо сейчас и когда лимит обновится.

### Чанкование

- Если исходный файл уже **< 19 MB** и формат поддерживается — отправляем **одним запросом**, без сегментации.
- Иначе режем через **ffmpeg** (`-f segment`).
- Длина чанка подбирается так, чтобы:
  1. Уложиться в 19 MB при заданном битрейте.
  2. Не превышать таймауты Groq.
- Базовая стратегия (как в LectureLog): сегменты по **1200 сек (20 мин)** при `libmp3lame -b:a 128k` ≈ 18 MB. На устройстве будем уточнять под реальный битрейт и при необходимости — пересжимать (downmix в моно, понижение sample rate до 16 kHz — для речи качество не страдает).

### Параллельная отправка

- Чанки отправляются параллельно. Ограничение через семафор: `Semaphore(min(len(alive_keys), N))`. Один чанк → один ключ → один in-flight запрос, остальные ключи в это время свободны под другие чанки.
- Ретраи на транзиентных ошибках с экспоненциальной задержкой (`5 * 2^attempt`).
- На 524 — ждать и повторить тот же чанк.

### Сборка результата

Groq возвращает `verbose_json` с пословной разметкой (`timestamp_granularities[]=word`). Каждое слово получает оффсет `index * chunk_duration` и складывается в общий список. На выходе:

- `transcript.srt` — субтитры с таймкодами.
- `transcript.txt` — сплошной текст (новое относительно LectureLog, нужно для подачи в LLM).

## Источники аудио

1. **Микрофон / запись в приложении** — пишем сразу в один из поддерживаемых форматов (предпочтительно `m4a`/AAC на Android, `wav` для совместимости).
2. **Локальный аудиофайл** — пользователь выбирает через `file_picker`. Валидируем расширение по белому списку.
3. **Локальный видеофайл** — выбор через `file_picker`. Отделяем аудиодорожку через ffmpeg: `ffmpeg -i input.mp4 -vn -acodec libmp3lame -b:a 128k out.mp3`. Видео не сохраняем.
4. **YouTube по ссылке** — скачиваем только аудио (см. ниже), дальше — как с локальным файлом.

## YouTube

- **Desktop:** `yt-dlp` как внешний бинарь (`-f bestaudio --extract-audio --audio-format mp3`).
- **Android (план — экспериментальный):** начинаем с чистого Dart-парсера (`youtube_explode_dart`) как самого лёгкого варианта. Реально протестировать на устройствах в разных сетях: возможно, что без VPN скачивание из РФ работать не будет. По итогам эксперимента примем решение — оставить как есть, добавить fallback на backend-прокси, или отказаться от YouTube на Android.
- Из загруженного аудио идём в общий пайплайн транскрибации.

## ffmpeg / ffprobe

- **Android/iOS:** `ffmpeg_kit_flutter` (FFmpegKit). Содержит и ffmpeg, и ffprobe.
- **Windows/desktop:** системный `ffmpeg.exe`/`ffprobe.exe` через `Process.run`. На первом запуске — проверка наличия, если нет — инструкция/автозагрузка.
- Использование:
  - `ffprobe` — длительность, битрейт, кодек до начала работы.
  - `ffmpeg` — сегментация, перекодирование, извлечение аудио из видео.

## Пайплайн (общий вид)

```
[источник] → [нормализация в поддерживаемый формат]
          → [ffprobe: метаданные]
          → [ffmpeg: сегментация ≤ 19 MB]
          → [Groq Whisper через KeyPool, параллельно]
          → [склейка слов с учётом offset]
          → [transcript.srt + transcript.txt]
          → [готов к передаче в LLM]
```

## Язык распознавания

- В UI — селектор языка с дефолтом «Авто». При «Авто» параметр `language` в запрос к Groq не передаётся, Whisper определяет сам.
- При явном выборе (например, `ru`, `en`) — передаём `language=<код>` в multipart-форме. Это повышает точность на «трудных» языках (русский, в частности).

## Хранение

- API-ключи — `flutter_secure_storage`.
- Промежуточные чанки — во временной директории приложения, удаляются после завершения задачи.
- Результаты (srt/txt) — в постоянном хранилище приложения, с возможностью экспорта.

## Что НЕ переносится из LectureLog

- FastAPI-сервер, CLI, Telegram-бот — на устройстве не нужны.
- Gemini-пайплайн структурирования и обработки слайдов — это уже отдельная LLM-стадия; для MVP ezctx достаточно сырой расшифровки. Возвращаемся к этому позже.
- Docker — нерелевантно.

## Ещё к проверке

- **Concurrency на одном ключе.** Пользователь предполагает, что Groq допускает несколько параллельных запросов с одного ключа (особенно когда дневная квота ещё не тронута). В коде это уже допустимо: семафор стоит на in-flight запросах, а не «по одному запросу на ключ». Но прежде чем агрессивно слать N запросов с одного ключа, нужно проверить на практике:
  - сколько одновременных in-flight Groq реально терпит без 429 (видимо, ограничение — RPM, а не «один запрос в моменте»);
  - имеет ли смысл при наличии нескольких ключей всё равно сначала ротировать по разным ключам, а уже потом наращивать concurrency на каждом.
  Решение по итогам экспериментов: настроить лимит in-flight per-key (вероятно 2–4) и общий лимит как `keys * per_key_limit`.
